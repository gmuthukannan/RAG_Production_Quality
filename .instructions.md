# .instructions.md

## Purpose

This repository builds a **production-quality RAG (Retrieval Augmented Generation) system**.

AI assistants working in this repo must preserve:
- Clear separation of ingestion, retrieval, and generation
- Deterministic, debuggable pipelines
- Minimal abstractions
- Predictable file structure

If any request conflicts with this file, this file wins.

---

## Project Architecture (MANDATORY)

This is a **pipeline-driven RAG system**.

Folders mean specific responsibilities and must never be mixed.
config/ → environment, secrets, runtime config
data/ → vector DB + documents
scripts/ → one-off ingestion and maintenance scripts
src/
components/ → small reusable RAG building blocks
pipelines/ → orchestration of components into flows
main.py → entry point only
utils.py → pure helpers, no RAG logic
tests/ → tests


---

## Mental Model

RAG here is strictly:
Loader → Chunker → Embedder → VectorStore → Retriever → Prompt → LLM


Each of the above is a **component**.

A pipeline wires components together.

AI must never collapse multiple stages into one file.

---

## Folder Rules (STRICT)

### `src/components/`

Contains ONLY single-responsibility units:

Examples:
- `loader.py`
- `chunker.py`
- `embedder.py`
- `vector_store.py`
- `retriever.py`
- `prompt_builder.py`
- `llm_client.py`

Rules:
- No orchestration
- No app logic
- No printing
- No CLI code

---

### `src/pipelines/`

This is where components are composed.

Examples:
- `ingestion_pipeline.py`
- `query_pipeline.py`

Rules:
- Pipelines call components
- Pipelines contain flow logic
- No low-level implementation here

---

### `scripts/`

Used for:

- document ingestion
- re-embedding
- DB reset
- maintenance

Scripts may import pipelines.

---

### `main.py`

Only does:

- Parse input
- Call the correct pipeline
- Print output

Nothing else.

---

## Coding Rules

AI must:

- Keep files under 250 lines
- Keep functions under 40 lines
- Use type hints everywhere
- Prefer explicit code over abstraction
- Use simple classes or functions, not frameworks

AI must NOT:

- Introduce new folders
- Merge components
- Add design patterns
- Add logging frameworks
- Add caching unless asked
- Add async unless asked

---

## Chroma Rules

Vector DB lives in: data/chroma_db


Rules:
- Collection name must be constant
- Embedding model name must be constant in one place
- Never recreate DB path dynamically

---

## Prompt Rules

Prompt construction must live in:components/prompt_builder.py


No prompts inside pipelines.

---

## LLM Rules

LLM access only via: components/llm_client.py


No direct OpenAI/LLM calls anywhere else.

---

## Utilities

`utils.py` is for:
- text cleaning
- small helpers
- file helpers

No RAG logic allowed.

---

## What AI Must Do When Asked to Add a Feature

Example: “Add metadata filtering”

AI must:
1. Modify `retriever.py`
2. Adjust `query_pipeline.py`
3. Not touch other files

---

## What AI Must NOT Do

AI must NOT:
- Put embedding code in pipelines
- Put retrieval code in main.py
- Add new abstraction layers
- Create a “services” folder
- Refactor the project structure

---

## Dependency Rules

Before adding any new library, AI must ask.

Allowed core stack:
- chromadb
- sentence-transformers / openai embeddings
- python stdlib

---

## Testing

Tests mirror pipelines and components.

No new test framework.

---

## Final Principle

This repo is a **teaching-quality RAG reference implementation**.

Clarity > cleverness.
Separation > convenience.
Pipelines > spaghetti code.

AI is an implementer, not a designer.
